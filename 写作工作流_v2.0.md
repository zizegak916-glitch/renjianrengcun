## 写作工作流

### 1. 正文写作

**每章结构：**
- 字数：2000-3300 字（批量输出 2200-2500 字）
- 每段 1-3 句
- 乐子人底色（陈亦视角：旁观者+找事找死+即时刺激）
- 反应词："有意思"、"行吧"、"来都来了"
- 陈亦用名不用"他"
- 破折号 ≤5/章
- 日常嵌进危机——日常不能打断危机动作

**身份线核心原则（v10.0）：**
- **删干净**"死过一回变了""旧反应对不上""变了但能用"等标签
- **不准**任何角色替陈亦解释身份
- **不准**"身份谜"本身成为悬念——默认陈亦就是沈怀
- 陈亦没沈怀记忆，但所有角色按沈怀对待他
- 每次旧身份压题：陈亦不知道答案，靠现场观察+反问+抓关键利益点接招
- 当场转成"谁要写死我、谁能害我、谁在试我、我拿什么活"

**身份线正确场景：**
| 场景 | 正确写法 |
|------|----------|
| 杜衡试暗语 | 接不上→反问"接错一句你就写我回死的？"→逼换格 |
| 小满问藏物 | 反问"你要的是能救人的，还是能害我的？"→当场判断 |
| 小吏质疑 | 怼"所以我以前死了"→反压谁换死因 |
| 小满葱碗试 | 夹错葱→抓饭量证词补回来 |
| 曹留听出不对 | 敲碗截话，硬吃饭+记路 |
| 刺客埋伏 | 不懂旧身位→躲偏反而活 |

### 2. 写后审查（三步）

**第一步：字数核验**
```
wc -m chapter.txt  # 中文实际字数
```

**第二步：禁用词扫描**
```
grep -E '冷损|胸冷|体弱|写没|候源|活人锚点|本章代价|候而未决' chapter.txt
grep -Ec '——' chapter.txt  # 破折号计数，不超过5
```

**第三步：元叙事审查**
无"我觉得""读者可能""这里需要铺垫"。如果写了不用改，直接删段。

**修词清单（按顺序）：**
1. `他→陈亦`（把开头的"他"改成"陈亦"）
2. `沈怀的记忆→删`（陈亦没有沈怀记忆，零）
3. `词汇替换`（见补丁集和风格文档）

### 3. 合并与发送

- Mobile TG：纯文本 + .txt 附件
- 输出目录为 `/root/人间仍存_写作输出/`

## Codex CLI 交互

Codex 用于大规模重构（生成/重写大纲、批量产出章节底稿）。多轮大纲修订工作流详见 `references/codex-outline-revision.md`。

**基本命令：**
```bash
cd /root/人间仍存/
codex exec --dangerously-bypass-approvals-and-sandbox -c model=gpt-5.5 "prompt"
```

**延续上次对话：**
```bash
cd /root/人间仍存/
codex exec resume --last --dangerously-bypass-approvals-and-sandbox -c model=gpt-5.5 "prompt"
```
`resume --last` 会载入最近一次 Codex 会话的上下文，用于迭代性修订（如大纲修改意见延续同一份大纲的 Codex 会话）。不需要传 session ID。

**超长 Prompt 处理（>3KB）：**
不要把超长修订意见塞进命令行参数——会触发 shell 长度限制或转义问题。写成文件，管道传入：
```bash
cd /root/人间仍存/
cat 修订意见文件.txt | codex exec resume --last --dangerously-bypass-approvals-and-sandbox -c model=gpt-5.5 -
```
末尾的 `-` 告诉 Codex 从 stdin 读取 prompt。运行方式：`terminal(background=true, notify_on_complete=true)`，Codex 需要几分钟处理大规模大纲文件。

**Git repo 要求：** Codex 必须在 git repo 内运行。本地 `/root/人间仍存/` 已初始化为 git repo。

**模型：** gpt-5.5（默认），不需要 OPENAI_API_KEY，需要 ChatGPT OAuth 认证。

**关键参数：**
- `--dangerously-bypass-approvals-and-sandbox`：在 Hermes 环境必须加（bubblewrap 失败）
- `-c model=gpt-5.5`：指定模型
- `resume --last`：延续最近一次 Codex 会话（用于迭代修订）


**工作流：**
1. 把源材料（原大纲/全文 + 修改意见 + 补丁集 + 角色内核）复制进 repo
2. 写 Prompt，让 Codex 读材料后重写
3. 后台跑，等完成
4. 从 rollout 或直接输出文件提取结果

**输出位置：**
- stdout 直接输出，部分写入 git repo 内
- 同时存到 rollout JSONL：`~/.codex/sessions/YYYY/MM/DD/rollout-<session-id>.jsonl`

**重要：**
- 给完修订意见后，Codex 自己生成文件，Hermes 只负责转交文件给用户，不参与任何其他环节（不审、不改、不评价、不总结）
- 超长 prompt 必须写文件 + 管道传入，不能塞命令行参数

## ⚠️ 接收用户内容：禁止消化/截断

当用户发送需要保存的内容（大纲、章节正文、修订意见等）时：
- **禁止消化** — 不要总结、不要改写、不要"提取要点"
- **禁止截断** — write_file 时不能出现 `...[truncated]`，必须写入完整内容
- **完整复制粘贴** — 用户给什么就原样写入什么
- 如果内容太长导致一次 write_file 写不下，分多次补全，但绝不能截断
- 写完后验证：`wc -l` 确认行数与用户提供的一致

**触发场景：** 用户说"复制粘贴"、发送大段文本要求保存、发送大纲/正文等创作内容。
